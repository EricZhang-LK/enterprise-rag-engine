# 为什么 RAG 项目要先做工程边界

很多 RAG Demo 都从类似流程开始：

```text
加载文档 -> 切分文本 -> 生成向量 -> 检索 -> 交给模型回答
```

这个流程足够做原型，但很难支撑企业级场景。

## 真正的问题

RAG 效果差时，很多人第一反应会怀疑模型、embedding 或向量数据库。但在真实项目里，
很多失败其实更早发生：

- 解析器丢了页码
- 表格结构被破坏
- chunk 脱离了章节标题
- 检索结果无法追溯到原始文件
- API 错误响应格式不统一
- 评测脚本无法复现 bad case

这些不是模型问题，而是工程边界问题。

## 工程边界带来什么

清晰的数据契约能让 RAG 系统更容易检查和改进。

`Document` 定义标准化后的文档是什么样。

`DocumentChunk` 定义什么内容可以被 embedding 和检索。

`ChunkMetadata` 保留来源、页码、章节、租户和 hash 信息。

`ParseResult` 明确解析成功、部分成功、失败、错误和耗时。

`RetrievalResult` 让排序、分数和检索器来源可见。

这些对象存在以后，每个模块的职责就会清楚很多。

## 为什么这对面试重要

面试官不只关心你会不会调用 embedding API。他们更想知道你能不能构建一个经得起生产
约束的系统：

- 失败能否追踪？
- 检索质量能否量化？
- 来源引用能否保留？
- 组件能否替换？
- 服务能否测试和部署？

先建立工程边界，会让你在这些问题上有更扎实的回答。

## W1 原则

不要急着做一个看起来很酷的 RAG Demo。

先建立契约。这样 parser、splitter、retriever、evaluator 和 API 才能在后面稳定生长，
不会变成一堆互相耦合的脚本。
